# v2.1 Phase 3-3 — `interrupt + resume` Deferred (Not a TODO)

> **Date**: 2026-04-29
> **Status**: Deferred by design — *not* an open TODO
> **Re-open trigger**: prod data, see "When to revisit" below
> **Predecessor**: `docs/v2.1-phase-3-active-compact-design.md`

This note exists so the next contributor doesn't pick Phase 3-3 up as
"unfinished work" and re-walk the same evaluation. The full
interrupt + resume orchestration is **deliberately not implemented**;
the monitor-only half (commit `e9e5f92`) is the current ship-state and
it is, today, sufficient.

## What is in production today

| Layer | Mechanism | Where |
|-------|-----------|-------|
| Compact detection | `compact_boundary` system message handler | `claudeRuntime.ts` |
| Compact recovery | `buildRecoveryNote()` with priority-budget + last-N raw tool preservation | `recoveryNoteBuilder.ts` (Phase 3-2) |
| Pre-rot pressure decision | Pure threshold function | `contextTokenMeter.ts` (Phase 3-1) |
| Pre-rot monitoring | One-shot warn + SSE progress when threshold crossed | `claudeRuntime.ts:checkContextPressure` (Phase 3-3 monitor) |
| Mid-execution restatement | `update_plan_phase` reminder + `fetch_artifact(full|rows)` reminder | `phaseHintMatcher.ts` + `activePhaseReminder.ts` (Phase 3-4) |

## Why we stopped here

The original Phase 3-3 design (`v2.1-phase-3-active-compact-design.md`)
was: when the token meter trips, call `query.interrupt()`, build a
recovery preamble from `buildRecoveryNote()`, and start a fresh
`query({ resume })` whose `prompt` carries the preamble + the
original user query.

Three reasons that's a bad bet right now:

### 1. Default budgets put us comfortably below the threshold

| Quantity | Value |
|----------|-------|
| `CLAUDE_MAX_TURNS` (full mode) | 30 |
| Typical per-turn uncached input | 500–2 000 tokens |
| Typical full-mode session uncached + creation total | ~60–90 K tokens |
| Pre-rot threshold (Phase 3-1 default) | 200 000 × 0.6 = 120 000 tokens |
| SDK auto-compact trigger | observed around 150–180 K tokens |

Only sessions with multiple `fetch_artifact(detail=full)` calls or
long tool-result payloads can plausibly push past 120 K. SDK
auto-compact handles the same ones a beat later, and our Phase 3-2
recovery note is already wired to that path.

### 2. The original pain point is already solved

Phase 3 existed because "after compact the agent didn't know what it
had just been doing". Phase 3-2 added a `recent_tool_calls` section
to the recovery note that carries the last N=5 structured tool-call
digests. The compact-then-resume flow now has a memory.

Phase 3-3 full only buys *earlier* recovery (one turn earlier than
SDK auto-compact). That is a marginal-quality gain whose cost is a
control-flow rewrite of the SDK stream loop.

### 3. The SDK behaviour for `interrupt()` mid-stream is undocumented

Open questions the design note flagged:

- Does an in-flight tool call cleanly error out, or does the SDK leave
  state in a half-applied form?
- Does `resume` re-send the whole prior history, or trust the
  server-side cache? Cost-wise the difference is large.
- Does the watchdog / circuit-breaker tick during the interrupt
  window? Risk of double-counting on resume.

Each unknown is a probable runtime regression. Without prod data
that says we *need* this lever, the expected value of pulling it is
negative.

## When to revisit (and what would change the calculus)

Re-open Phase 3-3 if **any** of the following holds at the next v2.x
review:

| Signal | Threshold | Action |
|--------|-----------|--------|
| `[ClaudeRuntime] pre-rot threshold crossed` log frequency | > 15 % of full-mode sessions over a 1-week window | Build the interrupt + resume prototype per design note |
| User-reported "compact 后丢细节" complaints | ≥ 3 distinct sessions citing it | Ditto |
| `CLAUDE_MAX_TURNS` raised above 50 | any time | Pre-rot probability climbs linearly — re-evaluate |
| SDK 1.x exposes `cache_control` *or* `pushMessage` | any time | Re-check the SDK spike doc; many of the "high-risk" parts disappear |

If none of those fires within 4 weeks, set
`CLAUDE_PRECOMPACT_WARN_ENABLED=false` and treat the monitor + meter
as utility libraries that ship in case a future feature wants them.

## What this note is *not*

- Not a roadmap promise. We may permanently retire the idea if data
  shows auto-compact + recovery note covers ≥ 95 % of sessions.
- Not a blocker for v2.2 / v2.3 work. The runtime is in steady state.
- Not a replacement for the design note. If we *do* start the work,
  re-read `v2.1-phase-3-active-compact-design.md` first; the 4-PR
  split there still applies.

## TL;DR

`Phase 3-3 full` is intentionally deferred behind a data trigger. The
monitor (Phase 3-3 monitor-only) is collecting the data. If the
trigger never fires, the project closes the loop without ever having
to write the risky orchestration code.
