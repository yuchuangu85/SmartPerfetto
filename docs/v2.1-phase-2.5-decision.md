## The three paths the v2.1 plan offered

| Path | Description |
|------|-------------|
| **A** | Backfill `phase_hints` to all 12 scenes |
| **B** | Drop `phase_hints` entirely; let `plan_template` do the work |
| **C** | Keep the 3 scenes that have hints; clarify the role split |

## Decision: **C** ‚Äî keep, clarify role

### Why not A

`plan_template` (Phase 2.1) already covers all 10 scenes that need a
hard-gate; backfilling `phase_hints` to every scene would either:

- duplicate the same constraints in two places (`plan_template` aspects
  vs. `phase_hints` constraints), creating a maintenance trap, or
- require synthesising shallow `phase_hints` for scenes (memory, game,
  overview, ‚Ä¶) whose strategies don't actually have phase-level depth
  worth restating

Either outcome adds bytes without adding signal.

### Why not B

The 3 scenes with `phase_hints` (scrolling, startup, anr) genuinely
need *mid-execution* restatement, not just *plan-submission* gating.
The two systems serve different lifecycle stages:

| System | When it fires | What it enforces |
|--------|---------------|------------------|
| `plan_template.mandatory_aspects` | At `submit_plan` / `revise_plan` (Phase 2.2+2.3 hard-gate) | The plan covers required analysis aspects |
| `phase_hints` | At `update_plan_phase` and `fetch_artifact(full|rows)` (Phase 3-4) | The agent stays on-strategy mid-analysis |

Dropping `phase_hints` would leave the mid-execution position empty.
The SDK spike confirmed we cannot push reminders mid-stream by other
means, so MCP tool responses are our only injection surface ‚Äî and
that's exactly what `phase_hints` is for.

### Why C

Keep the existing 3-scene coverage; do **not** force-backfill the
remaining 9. Instead, document the role split (above table) so that
future scene additions know which knob to reach for:

- New scene needs hard-gate? ‚Üí add `plan_template.mandatory_aspects`.
- New scene has multi-phase mid-execution drift risk? ‚Üí also add
  `phase_hints`.
- Otherwise? ‚Üí neither.

This is **exactly the steady state we already have**. The decision
costs zero code; the value is preventing future contributors from
re-running this same evaluation.

## Acceptance signal for revisiting

Re-evaluate this decision if either of the following holds at the next
v2.x review:

1. `phaseHintMatcher` "phase hint not found" log entries exceed 30%
   of `update_plan_phase` calls in prod (suggests some scenes need
   `phase_hints` after all).
2. A new scene lands that has multi-phase strategy depth comparable
   to scrolling / startup / anr.

Until then, the 3-scene asymmetry is a feature, not a bug.
